## 1. 概述

测试是现代软件开发过程的重要阶段。 无论采用何种开发方法，无论是传统方法(例如级联)还是敏捷方法(例如[极限编程](https://www.baeldung.com/cs/simple-design-clean-code#the-extreme-programming-methodology))，测试都是流程中必不可少的部分。

但是，并非所有测试都是相同的。存在多种测试类别。这些类别区分，例如，测试如何在软件中运行(黑盒和白盒)或在软件中测试哪些(单元、集成、系统和验收)。

在本教程中，我们将专门研究黑盒和白盒测试。 首先，我们将简要介绍软件开发中的测试。因此，我们将专门研究什么是白盒和黑盒测试以及它们如何工作。最后，我们将概述探索的概念并在系统总结中比较白盒和黑盒测试。

## 2. 软件开发中的测试

测试包括评估软件程序功能的过程。 测试过程的主要目标是检查开发的软件是否按预期运行。

此过程通常检查低级单元，检查它们是否按照其技术规范实施；以及将这些单位整合成一个完整的系统，分析它们如何合作。此外，开发人员还测试高级例程，作为开发系统的最终用户。

从技术上讲，我们可以手动执行测试或将它们编程为自动执行。 使用自动化测试的好处是它们可以作为已实现功能及其在系统中正确操作的清单。一些开发方法，例如测试驱动开发 (TDD)，甚至在开始对目标系统本身进行编程之前就考虑创建自动化测试。

在开发过程中测试软件/系统有四个明显的好处：

-   成本效益：从长远来看，测试有助于节省资金，因为与在后期开发阶段或发布后发现错误或不兼容相比，在软件开发过程的早期阶段发现错误或不兼容性的解决成本要低得多
-   安全改进：测试提高了系统的可信度和安全性。执行它们可以降低已发布软件中存在后门、错误或其他漏洞的风险
-   高质量产品：执行测试可避免发布功能损坏或配置错误的软件或系统
-   高客户满意度：采用 UX/UI 驱动的测试来分析用户体验，根据结果调整软件，是提高对已发布系统的整体满意度的一种方式

考虑到所有呈现的背景概念，我们将在以下部分深入探讨白盒和黑盒测试。

## 3. 白盒测试

白盒测试是一种探索算法的输入、输出和处理流程的技术。 这意味着白盒测试需要访问源代码：测试考虑给定软件的操作例程的实现细节。因此，白盒测试也称为透明盒测试、玻璃盒测试和基于代码的测试。

简而言之，白盒测试考虑了软件的内部视角，通常只有开发团队才能访问。

白盒测试评估软件的几个方面。让我们看看其中的一些：

-   执行流程(根据输入)
-   软件结果(作为输出)
-   潜在的安全漏洞
-   可能损坏的数据结构
-   循环停止标准
-   每个已实现操作(方法或函数)的功能

因此，准备和执行白盒测试例程需要有关被检查软件的大量知识。除了了解内部流程外，测试人员还必须定义预期结果并验证它们是否与获得的结果与测试相匹配。

白盒测试的相关优势是通过结果实现代码优化、轻松实现测试自动化，以及可以测试算法的所有操作路径，甚至包括最终用户无法使用的路径。但是，它们通常既昂贵又复杂，只有参与软件开发过程的人员才能有效地创建和执行它们。

有多种白盒测试技术。我们将在以下小节中简要研究其中的一些。

### 3.1. 报表覆盖率

语句覆盖背后的主要思想是执行特定算法的每个语句。因此，语句覆盖测试可能需要在不同的场景和异构输入中多次执行相同的代码。

在分支覆盖技术中，采用算法中每个决策点的每条路径。同样，它可能需要多次执行相同的算法(可能具有不同的输入)才能完成分支覆盖测试。

## 4. 黑盒测试

黑盒测试是一种通过与代码无关的角度来分析软件或系统的技术。这意味着测试人员无权访问组成测试软件的算法(源代码)，通常通过某些界面对其进行操作。由于这些特点，我们也可以把这样的测试过程称为行为测试。

总之，黑盒测试从软件的外部角度来看，关注给定特定输入的输出。

存在几种类型的黑盒测试，但关注软件功能和非功能需求的是最相关的：

-   功能测试：目的是验证软件是否以正确的方式完成为其定义的所有功能要求
-   非功能测试：目标包括分析测试软件是否具有足够的性能、可扩展性和可用性，以及其他非功能需求
-   回归测试：目的是保证添加到软件中的新功能与已有的其他功能完全兼容，不会给它们带来任何问题

因此，要执行黑盒测试，测试人员必须对软件规范有很多了解，从而能够评估特定输入的结果是否符合预期(并令人满意地生成)。

黑盒测试的一些优点是测试人员不需要太多低级技术知识来执行它们(我们使用最终用户的角度来完成测试)。此外，这些测试通常比白盒测试便宜并且易于重现。但是，黑盒测试可能无法揭示失败的原因，我们可能无法检测到软件控制结构中的错误。

让我们在以下小节中简要探讨一些最知名的黑盒测试技术。

### 4.1. 等价类测试

一种识别在提交以测试软件和系统的特定功能时返回相同结果的输入的技术。因此，我们的想法是对这些输入进行聚类。然后，我们考虑每个生成的集群的样本而不是所有输入可能性来组织测试。

### 4.2. 边界值测试

边界值测试对于测试接收一系列值输入的函数特别有用。因此，这里的想法是创建一组具有此类范围边界值的测试：最小值和最大值。

下图显示了边界值技术的示例：

### 4.3. 因果图

因果图建立了提供的输入和结果输出之间的关系。在这种情况下，我们将输入视为原因，将通过提供一组输入捕获的输出视为结果。

主要思想是将输入连接到因果图中的输出。接下来，将因果图转化为决策表，用于为待测软件或系统创建一组具有代表性的测试用例。

## 五、系统总结

正如我们在前面几节中看到的，测试对于保证具有成本效益的编程项目、具有操作安全性的高质量软件产品以及高水平的客户满意度至关重要。

有许多类别的测试。特别是，白盒和黑盒测试是指我们如何解释软件和系统以对其执行测试。

白盒测试深入研究软件源代码中定义的处理流程。然而，黑盒测试与源代码无关，它通过给定一组输入的操作行为来分析软件。

因此，白盒测试与开发人员对软件和系统的看法密切相关。另一方面，黑盒测试假设的视角更接近软件程序的最终用户。
